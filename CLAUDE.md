# Banking Api — CLAUDE.md

## Project

Spring Boot 3.5 banking REST API. Stateless authentication via JWT stored in an HttpOnly cookie. Serves an Angular 21 frontend.

## Stack

- **Framework**: Spring Boot 3.5 / Java 21
- **ORM**: Spring Data JPA + Hibernate → PostgreSQL (dev: `localhost:5432/banking`)
- **Security**: Spring Security + JJWT 0.12.3 (HttpOnly cookie)
- **Validation**: Bean Validation (`@Valid`, `@NotBlank`, `@Positive`…)
- **Docs**: SpringDoc OpenAPI 2.8.6 (Swagger UI at `/swagger-ui.html`)
- **Build**: Maven

## Architecture

```
src/main/java/com/mehdi/banking_api/
├── config/           # OpenAPI config
├── security/         # JwtService, JwtAuthFilter, SecurityConfig
├── model/            # JPA entities + enums (AccountType, TransactionStatus)
├── repository/       # Spring Data repositories (JpaRepository)
├── service/          # Business logic
├── controller/       # REST controllers
├── dto/
│   ├── request/      # Input DTOs (validated)
│   └── response/     # Output DTOs
├── exception/        # Custom exceptions + GlobalExceptionHandler
└── common/
    └── ApiResult.java # Generic response wrapper
```

## Security — critical rules

### JWT cookie
- Token generated by `JwtService`, stored as an HttpOnly cookie named **`jwt`**
- **Cookie attributes**: `httpOnly=true`, `secure=true`, `sameSite=None`, `domain=.mdiasgomes.com`, `maxAge=86400`
- `JwtAuthFilter` reads the cookie on every request, validates it, and populates the `SecurityContext`
- No `Authorization: Bearer` header — authentication is entirely cookie-based

### Public endpoints (no auth required)
```
/api/auth/**
/error
/v3/api-docs/**
/swagger-ui/**
```
All other routes require a valid `jwt` cookie.

### CORS allowed origins
```
http://localhost:4200
https://banking-frontend-beta.vercel.app
https://banking.mdiasgomes.com
```
`credentials: true` is mandatory because of the cookie.

## Response shapes

### `ApiResult<T>` wrapper (used by `/transactions/**`)
```java
record ApiResult<T>(
    boolean success,
    int     status,
    String  message,
    T       data,
    LocalDateTime timestamp
) {}
```
Build with the static factory: `ApiResult.success(data)`.

### Direct body (used by `/api/**`)
Controllers under `/api/auth`, `/api/users`, `/api/accounts`, `/api/beneficiaries` return the DTO directly — **no wrapper**.

> **Do not mix the two shapes.** When adding a new endpoint decide which group it belongs to and be consistent.

## Entities

| Entity | Key fields |
|--------|-----------|
| `User` | `id: UUID`, `email`, `password` (BCrypt), `firstName`, `lastName` |
| `Account` | `id: UUID`, `iban` (LU format, auto-generated), `balance: BigDecimal(19,4)`, `type: AccountType`, `owner: User` |
| `Transaction` | `id: UUID`, `amount`, `sender/receiver: Account`, `status: TransactionStatus`, `createdAt` |
| `Beneficiary` | `id: UUID`, `iban`, `bankCode`, `countryCode`, `firstName`, `lastName`, `owner: User`, `createdAt` |

**Enums**: `AccountType { CHECKING, SAVINGS }` · `TransactionStatus { PENDING, COMPLETED, FAILED }`

## Monetary amounts

`balance` and `amount` are `BigDecimal` with precision 19, scale 4. Never use `double` or `float`.

## Exception handling

All exceptions are caught by `GlobalExceptionHandler` and serialized as `ApiResult<?>`:

| Exception | HTTP |
|-----------|------|
| `ResourceNotFoundException` | 404 |
| `BusinessException` | 400 |
| `ForbiddenException` | 403 |
| `ApiException` | custom status |
| `DataIntegrityViolationException` | 409 |
| `MethodArgumentNotValidException` | 400 + field map |

Always throw the most specific exception. Never return raw error strings from controllers.

## Java conventions

- Lombok (`@Data`, `@Builder`, `@RequiredArgsConstructor`) everywhere — no manual getters/setters
- Services are `@Transactional` at class level; use `@Transactional(propagation = REQUIRES_NEW)` for audit writes
- Controllers only orchestrate: extract the authenticated user from `SecurityContext`, call one service method, return the result
- No business logic in controllers or repositories
- DTOs are separate from entities — never expose JPA entities directly in responses
